# Pattern C：重构

适用于：对现有代码进行结构性重构，不改变外部行为

---

## 使用场景

- 提取重复代码为公共模块
- 分层重构（将业务逻辑从控制器移至服务层）
- 性能优化（改写查询、缓存策略）
- 技术债清理

## 与 Pattern A 的核心区别

重构的 AC 是**行为不变**，而不是新增行为：

```markdown
## Acceptance Criteria (AC)
- AC-1: 所有现有测试继续通过（行为不变）
- AC-2: 重构后的代码消除了 X 处重复
- AC-3: 方法行数从 N 行降至 M 行以下
```

## 流程

```
spec.md（描述重构目标和约束）
  │
  ├─ /sdd:plan      猫头鹰分析现有代码，制定重构策略
  │                 （重点：Phase -1 反过度抽象门禁）
  │
  ├─ /sdd:tasks     啄木鸟拆解（先写测试 → 再重构 → 再验证）
  │
  └─ /sdd:implement 海狸重构（"先红后绿"原则：先确保测试覆盖，再重构）
```

## plan.md 特殊要求

重构方案必须包含：

1. **现状分析**：描述当前代码的问题（带文件路径）
2. **重构策略**：选择的重构手法（提取方法、移动职责、引入接口等）
3. **风险评估**：可能影响的调用方
4. **回滚方案**：如何撤销

## 注意事项

- 重构前先补充测试覆盖（如果已有测试不足）
- 每次 Task 只做一种重构操作，保持最小变更
- contracts/ 通常不需要更新（外部接口不变）
- constitution.md 的"简洁优于抽象"原则在此尤为重要

## 禁止的重构模式

- 不要为"将来的灵活性"引入接口/抽象层
- 不要在重构时顺手添加新功能（分开做两个 Spec）
- 不要因为"优雅"而增加代码复杂度
