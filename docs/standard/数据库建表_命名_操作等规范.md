# MySQL 数据库开发规范（全文版）

> **来源**：哔哩哔哩内部文档  
> **版本**：2025-11-12 最后更新  
> **适用范围**：所有使用 **InnoDB** 存储引擎的 MySQL 数据库  
> **目的**：保障数据库 **性能、稳定性、可维护性、可扩展性**

---

## 🔴 重要提示（必须遵守）

| 项 | 要求 |
|---|---|
| 设计评审 | 所有 DB 设计必须经过内部评审；单个团队 3 次不合规 → DBA 拒绝处理 |
| SQL 优化 | 上线前必须用 `EXPLAIN` 或 Akso「优化建议」检查是否使用索引 |
| 大表操作 | 预留足够时间，部分操作可能耗时数小时 |
| 存储引擎 | 所有规范基于 **InnoDB** |
| 非标准库 | 不符合建表/授权等规范的库，无法保障性能与可用性 |

---

## 1️⃣ 数据库设计规范

### 1.1 必须遵守（强制）

| 维度 | 规范 |
|---|---|
| 命名 | 库、表、字段、索引 **全部小写**；禁止 MySQL 关键字与保留字 |
| 命名 | 所有的数据库、表除特殊情况外(表情支持等)，都不需要手动指定字符集 例：utf8mb4 |
| 主键 | 必须有 `BIGINT UNSIGNED NOT NULL AUTO_INCREMENT` 主键；<br>预估 ≤ 42 亿行用 `INT UNSIGNED`，> 42 亿行用 `BIGINT UNSIGNED` |
| 字段 | 全部 `NOT NULL DEFAULT`；<br>数值默认 `0`，字符默认 `''` |
| 时间字段 | 必须包含 `ctime DATETIME DEFAULT CURRENT_TIMESTAMP` 与 `mtime DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP` |
| 注释 | 每张表、每个字段必须写 `COMMENT` |
| 索引命名 | 非唯一索引 `ix_<字段名>`，唯一索引 `uk_<字段名>` |
| JOIN | 用于连接的字段 **类型完全一致** 并建索引 |
| 单实例 | 禁止多业务共用同一数据库 |
| 必建索引 | 列'mtime' 必须建索引，命名为ix_mtime |
| 排序规则 | 禁止设置排序规则 |

> **禁止使用 `NULL` 的原因**：索引效率低、存储额外开销、逻辑判断复杂、易出 Bug。

### 1.2 强烈建议

| 维度 | 建议 |
|---|---|
| 字符串 | `VARCHAR(N)` 在满足业务前提下 **N < 255**；节省磁盘与内存 |
| 枚举 | 用 `TINYINT` + `COMMENT` 替代 `ENUM`，避免 DDL |
| 排序 | **不要在数据库做大排序**；量大时在程序端处理 |
| 删除 | 采用 **逻辑删除**（`is_deleted` 标识），避免物理删除，纯log日志类型表除外 |
| 数据量级 | 单表控制在 **千万级** 以内；超量考虑归档或分表 |
| 组合索引 | 区分度高的字段放最左前缀 |

### 1.3 尽量避免

| 维度 | 原因 |
|---|---|
| `BLOB/TEXT/BINARY` | 浪费磁盘/内存，易淘汰热数据，导致缓存命中率下降 |
| 浮点型 | 计算精度误差；可用整型放大倍数 |
| 数据库计算 | 尽量把计算放到程序端，减轻 DB 压力 |
| 多表 JOIN | 尽可能单表查询，降低复杂度和锁竞争 |
| 关键字 | 禁止用 MySQL 关键字/保留字 |
| 冗余索引 | 如已有 `(a,b)`，则 `(a)` 为冗余 |

### 1.4 绝对禁止

| 维度 | 说明 |
|---|---|
| 删除字段/表 | 生产环境禁止 `DROP COLUMN/TABLE`；只允许 `ADD COLUMN` |
| `INSERT INTO SELECT` | 会锁表 |
| `UPDATE/DELETE ... LIMIT` | 无法确定影响行数；必须加 `ORDER BY` |
| 多表 JOIN | 禁止 **>2 张表** JOIN |
| 子查询 | 禁止子查询，可改写成 JOIN 或程序端分批 |
| 视图/存储过程/触发器 | 禁止使用 |
| 外键 | 禁止使用，逻辑由程序控制 |
| `ORDER BY RAND()` | 性能极差 |
| 字段顺序修改 | 禁止 `AFTER/BEFORE`，避免数据偏移 |

## 2️⃣ SQL 语句书写规范

### 2.1 CREATE TABLE

- 必须指定 `ENGINE=InnoDB`、表级/字段级 `COMMENT`
- 必须包含 `ctime`、`mtime`
- 主键必须为 `BIGINT UNSIGNED NOT NULL AUTO_INCREMENT`

**示例：**
```sql
CREATE TABLE `cloud_bill_analyze_record` (
`id` bigint unsigned NOT NULL AUTO_INCREMENT COMMENT '自增主键ID',
`cloud_id` varchar(32) NOT NULL DEFAULT '' COMMENT '云平台id',
`cloud_account` int NOT NULL DEFAULT 0 COMMENT '云平台云账户',
 `is_deleted` tinyint NOT NULL DEFAULT 0 COMMENT '是否被删除',
`cloud_account_name` varchar(64) NOT NULL DEFAULT '' COMMENT '云平台云账户名',
`collection_time` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '采集时间',
`ctime` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
`mtime` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '修改时间',
PRIMARY KEY (`id`),
KEY `ix_mtime` (`mtime`)
) ENGINE=InnoDB COMMENT='云账单分析记录';
```

### 2.2 ALTER TABLE 规范

| 操作 | 是否允许 | 备注 |
|------|----------|------|
| 新增字段 | ✅ | 禁止 `AFTER/BEFORE` |
| 修改字段类型 | ❌ | 仅允许 `INT→BIGINT` 或长度调整 |
| 修改字段名 | ❌ | 禁止 |
| 删除字段 | ❌ | 禁止 |
| 修改字段顺序 | ❌ | 禁止 |

### 2.3 索引操作规范

- **普通索引**：`ALTER TABLE tb ADD INDEX ix_col(col);`
- **唯一索引**：`ALTER TABLE tb ADD UNIQUE INDEX uk_col(col);`
- **删除索引**：`ALTER TABLE tb DROP INDEX ix_col;`
- **联合索引**：区分度高的放最左前缀

### 2.4 SELECT 规范

- 禁止 `SELECT *`
- 禁止隐式转换（`varchar = 123`）
- 禁止 `%前缀` 模糊查询
- 必须 `ORDER BY ... LIMIT`，禁止无序 `LIMIT`

### 2.5 INSERT 规范

- 必须显式指定字段名
- 批量插入 ≤ 200 条/次

### 2.6 UPDATE 规范

- 必须带 `WHERE`
- 禁止 `UPDATE ... LIMIT`
- 大批量更新按主键分批，每批 1k~3k 条，sleep 1~3s，循环提交

### 2.7 DELETE 规范

- 必须带 `WHERE`
- 禁止 `DELETE ... LIMIT`
- 建议逻辑删除；物理删除按主键分批

### 2.8 其他书写规范

- 禁止在字段上使用函数（`WHERE unix_timestamp(col)=xxx`）
- 禁止隐式类型转换（`varchar = 123` / `int = '123'`）
- 字段放操作符左边（`WHERE col = value`）

## 3️⃣ 程序端操作规范

### 3.1 必须遵守

- 长连接必须带 **自动重连** 与 **连接超时回收**（建议 2 h）
- 程序访问数据库连接的字符集请设置为 **utf8mb4**

### 3.2 绝对禁止

- 程序中执行任何 **DDL** 语句

---

## 4️⃣ 行为规范

- 禁止用配置文件里的账号 **手动访问线上库**
- 大型活动（拜年祭、抽奖）**提前与 DBA 做流量评估**
- 批量清洗数据 **避开业务高峰**，执行中观察负载
- 禁止在主库执行 **后台/统计类大查询**

## 5️⃣ 分库分表命名规则

| 策略 | 命名范例 |
|------|----------|
| 数字分表 | `table_000` ~ `table_999`（补零） |
| 年分表 | `table_2017` ~ `table_2025` |
| 月分表 | `table_202301` ~ `table_202512` |
| 日分表 | `table_20230101` ~ `table_20231231` |

## 6️⃣ 常用字段类型范围

| 类型 | 范围 |
|------|------|
| `TINYINT` | -128 ~ 127 |
| `TINYINT UNSIGNED` | 0 ~ 255 |
| `INT` | -21 亿 ~ 21 亿 |
| `INT UNSIGNED` | 0 ~ 42 亿 |
| `BIGINT` | 极大 |
| `VARCHAR(N)` | N 个字符（中英文均算 1） |
| `DATETIME` | 1000-01-01 ~ 9999-12-31 |
| `TIMESTAMP` | 1970-01-01 ~ 2038-01-19（有时区转换） |

## 8️⃣ 注意事项

### 8.1 数据修改

- 平台自动备份，可回滚
- **单条 SQL 扫描行数 > 5W 会被拒绝**（可用 `EXPLAIN` 预判）

### 8.2 批量数据处理

- 一个工单 ≤ 200 条 SQL
- 按主键分批

**示例：**
```sql
DELETE FROM test_table WHERE id BETWEEN 1 AND 1000 AND state = 9;
DELETE FROM test_table WHERE id BETWEEN 1001 AND 2000 AND state = 9;
```

### 8.3 大表 DDL

- ≥ 100 MB 的表用 **GH-OST** 在线改表，会产生临时表拷贝
- **QPS > 3000** 的实例避免高峰提交大表 DDL
- **≥ 2000W 行的表 DDL 不会自动执行**，需 DBA 人工评估

## 9️⃣ 常见问题 Q&A

| 问题 | 解答 |
|------|------|
| 默认值 & NOT NULL | 所有字段必须设置默认值并 NOT NULL |
| 关键字 | 禁止使用 MySQL 关键字/保留字 |
| DELETE 权限 | 默认不给，建议逻辑删除；必须用时建库备注 |

## ✅ 十句话背下开发铁律

1. **主键**必须是 `BIGINT UNSIGNED NOT NULL AUTO_INCREMENT`
2. **所有字段**必须 `NOT NULL DEFAULT`
3. **必有** `ctime`、`mtime` 且默认 `CURRENT_TIMESTAMP`
4. **禁止** `SELECT *`、`UPDATE/DELETE ... LIMIT`
5. **禁止**子查询、>2 表 JOIN、外键、视图、触发器
6. **禁止**在字段上使用函数或隐式转换
7. **大表结构变更**走 **GH-OST**，禁止高峰提交
8. **程序端禁止**执行任何 **DDL**
9. **所有变更**必须走 **Akso 平台**，禁止手动线上操作
10. **索引命名**：`ix_<字段名>` / `uk_<字段名>`；组合索引区分度高的放左边
